## Exploration: alternative base models / APIs for the translation step

I researched the candidates named in this issue (GLM-5.2, Cohere Command A) plus the open-source landscape, and drafted a pluggable-backend design so any of them can be slotted in without rewriting the pipeline. Summary below; the full write-up + a reference interface are prepared as `docs/translation-backends.md`, `processors/backend.py`, and `tests/test_backend.py` (not yet pushed — sharing findings here first).

### Context
Today the pipeline wraps a single hard-coded backend — **LINDAT/CUBBITT** bilingual NMT (`processors/translator.py`). Two consequences motivate this exploration:
1. **No native terminology control** → the brittle *Tag-and-Protect* workaround (UDPipe lemmatisation + placeholder sentinels) exists only to honour vocabulary terms.
2. **Licensing lock-in** → `para_licenses.py` propagates the most-restrictive component license; CUBBITT + UDPipe force every output to **CC BY-NC-SA 4.0**.

### Candidate comparison

| Backend | Type | Deploy | Languages (vs repo set) | Native glossary | License (weights / API) | Approx. cost |
|---|---|---|---|---|---|---|
| **LINDAT/CUBBITT** (baseline) | Bilingual NMT | Free UFAL API | cs + major EU pairs | No | CC BY-NC-SA 4.0 | Free |
| **Cohere Command A Translate** | Translation LLM (111B) | API or self-host (1–2× A100/H100) | 23 incl. cs, uk, pl, ru, de, fr, es, it, nl, ro, el, hi | **Yes** + doc mode | HF weights **CC-BY-NC** / Cohere API (paid) | per-token / GPU |
| **GLM-5.2** (Zhipu/Z.ai) | General LLM (744B MoE) | API only (self-host impractical) | Broad, en/zh-centric | Prompt-only | Weights **MIT** / API ~$1.40·$4.40 per M tok | ~$1–4 / M tok |
| **MADLAD-400** (Google) | Multilingual NMT (T5, 3–10B) | Self-host (3B CPU-viable) | 400+ incl. **all** repo langs | No | **Apache-2.0** | Infra only |
| **NLLB-200** (Meta) | Multilingual NMT (0.6–3.3B) | Self-host | 200+ incl. all | No | CC-BY-NC-4.0 | Infra only |
| **Tower+ 9B/72B** (Unbabel) | Translation LLM (Qwen2.5) | Self-host GPU | 22 incl. cs | Instructable | Research / Qwen terms | GPU infra |
| **Opus-MT** (Helsinki-NLP) | Bilingual NMT (Marian) | Self-host (CPU-viable) | Many pairs incl. cs-en | No | **Apache-2.0** | Minimal |
| **DeepL API** | Commercial NMT/LLM | API (EU) | European incl. cs (top quality) | **Yes** | Proprietary, paid | ~$25/M chars |
| **Google Cloud Translation** | Commercial NMT + LLM | API | Broad | Yes | Proprietary, paid | ~$20/M chars |

**Language coverage vs the repo's identifier** (cs, en, fr, de, ru, pl, uk, sk, bg, hr, sl, lv, lt, et, hu, ro, es, it, nl, hi): MADLAD-400 / NLLB-200 / Google cover all; Command A covers ~15/20 (missing sk, bg, hr, sl, lv, lt, et, hu) → those pairs would fall back to CUBBITT/MADLAD.

### Two cross-cutting findings
- **Terminology opportunity:** glossary-native backends (Command A, DeepL, Google) could retire the *entire* Tag-and-Protect machinery — UDPipe calls, placeholder sentinels, fuzzy restoration, the number-agreement guard, and the homonym-misalignment risk all disappear; the vocabulary CSV is passed straight to the backend's glossary API.
- **Licensing is a lever, not just a constraint:** to reach a *permissive* output license, both the backend **and** UDPipe must be replaced. A glossary-native backend removes UDPipe in the same move — e.g. MADLAD-400 (Apache-2.0) self-hosted, or GLM-5.2 (MIT), can yield permissive outputs.

### Recommendation
1. **Prototype first — Cohere Command A Translate:** best language/domain fit for cs→en archival text, native glossary (retires Tag-and-Protect), available as both API and self-host weights.
2. **Permissive self-host alternative — MADLAD-400 (Apache-2.0):** path to commercially-licensed output + broadest coverage; best if NC becomes a blocker or offline deployment is needed.
3. **API generalist — GLM-5.2 (MIT):** if ATRIUM wants one frontier LLM across NER/summarisation/translation; needs OCR-faithfulness guardrails + prompt-based terminology (less reliable, hallucination risk on noisy OCR).
4. **Keep CUBBITT as the default** — free, EU-hosted, proven; alternatives are opt-in.

### Proposed design (so this stays easy to extend)
A minimal `TranslationBackend` Protocol whose only required method is `translate(text, src_lang, tgt_lang="en")` — the single coupling point used across `utils.py`, `service/api.py`, and `main.py`. A `get_backend(name)` factory (env `TRANSLATION_BACKEND`, default `"lindat"`) returns the implementation. `LindatTranslator` satisfies the protocol structurally, so **no changes to `translator.py`** and existing behaviour is untouched (178/178 tests pass).

### Suggested next steps
1. Land the `TranslationBackend` seam.
2. Implement one reference adapter (Command A recommended) behind `TRANSLATION_BACKEND=cohere`.
3. Small quality bake-off on sample AMCR + ALTO data (CUBBITT vs adapter, ±glossary; BLEU/COMET + manual terminology review).
4. Decide default vs opt-in from results.

# Translation backends (issue #4)

Status: **in progress** — Phase 0 (seam) and Phase 1 (free LLM API adapter) implemented;
Phase 2 (bake-off harness) and Phase 3 (CTranslate2 self-host) scaffolded.

This document is the design reference for making the translator's **base translation
model pluggable**. It is referenced from `processors/backend.py`,
`processors/translator.py` and `tests/test_backend.py`, and it records: the backend
contract, how a backend is selected, the candidate comparison, the licensing recipe, and
the evaluation plan.

> Motivation (issue #4): explore base models — GLM, Cohere Command A, and the open-source
> space — to supplement or replace the default CUBBITT/LINDAT backend. The current
> priority is **free + low-resource** (both compute footprint *and* language coverage),
> prototyping first through a **free LLM API tier**, while keeping CUBBITT as the default
> so existing behaviour is unchanged.

---

## 1. The contract: `TranslationBackend`

A backend is any object satisfying the `@runtime_checkable` Protocol in
`processors/backend.py`:

```python
class TranslationBackend(Protocol):
    name: str                      # registry id, e.g. "lindat"
    supports_glossary: bool        # True  -> backend handles terminology itself
                                   # False -> pipeline uses Tag-and-Protect instead
    def translate(self, text: str, src_lang: str, tgt_lang: str = "en") -> str: ...
    def supported_languages(self) -> list[str]: ...
```

`translate()` is the **single coupling point** the rest of the pipeline depends on
(`utils.process_alto_xml`, `utils.process_metadata_xml`, `service/api.py`,
`main.process_single_file`). Because the Protocol is `@runtime_checkable` and declares
data members, `isinstance(obj, TranslationBackend)` also checks that `name` and
`supports_glossary` exist, so every backend must expose them.

### Pipeline-compatibility surface (beyond the Protocol)

`main.process_single_file` additionally calls a small surface that is **not** part of the
Protocol but every backend must provide to be a drop-in:

| Member                                                                | Purpose                                                                                                                                                                                  |
|-----------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `vocabulary: dict`                                                    | Non-empty when a glossary/vocab is loaded; gates per-document term stats.                                                                                                                |
| `reset_protected_count()`                                             | Zero the per-document protected/injected-term tally.                                                                                                                                     |
| `protected_count: int` (property)                                     | Read the tally after a document for paradata stats.                                                                                                                                      |
| `license_components(vocab_loaded: bool) -> list[str]` *(recommended)* | Component names (keys in `para_config.txt`) the backend actually exercised, so paradata licensing is correct after a swap. If absent, `main.py` falls back to the historical LINDAT set. |

Both shipped backends implement all of these.

---

## 2. Selecting a backend

Resolution order (first non-empty wins), implemented in `main.parse_arguments`:

1. `--backend <name>` on the CLI;
2. `translation_backend = <name>` in `config.txt`;
3. the `TRANSLATION_BACKEND` environment variable;
4. fallback `"lindat"`.

`service/api.py` reads `TRANSLATION_BACKEND` at warm-up. `get_backend(name, **kwargs)`
forwards constructor kwargs (e.g. `vocab_path`) to the selected backend and raises
`ValueError` for unknown names.

```bash
# default (no behaviour change)
python main.py ./data_samples/my_documents --alto --formats alto.xml

# free LLM API backend
export LLM_BASE_URL=https://openrouter.ai/api/v1
export LLM_MODEL="meta-llama/llama-3.3-70b-instruct:free"
export LLM_API_KEY=sk-...
python main.py ./data_samples/my_documents --alto --formats alto.xml --backend openai_compatible
```

## 3. Shipped backends

### 3.1 `lindat` (default) — `LindatTranslator`
LINDAT/CLARIAH-CZ CUBBITT. `supports_glossary = False` (no glossary API → terminology is
handled by the in-process **Tag-and-Protect** pipeline + UDPipe). cs-centric; non-`*-en`
pairs are silently rerouted to `cs-en`. Licence: CC BY-NC-SA.

### 3.2 `openai_compatible` — `LLMTranslator` (Phase 1, the prototype)
One adapter for **many** free/low-cost providers via any `/v1/chat/completions` endpoint —
no vendor SDK, only `requests` (extra deps in `requirements-llm.txt`).

| Env var                                                               | Meaning                                                                                                                                                                                                             |
|-----------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `LLM_BASE_URL`                                                        | Endpoint base, e.g. `https://openrouter.ai/api/v1`, `https://generativelanguage.googleapis.com/v1beta/openai`, `https://api.mistral.ai/v1`, `https://api.groq.com/openai/v1`, `http://localhost:11434/v1` (Ollama). |
| `LLM_MODEL`                                                           | Model id served there.                                                                                                                                                                                              |
| `LLM_API_KEY`                                                         | Bearer token (omit for keyless local servers).                                                                                                                                                                      |
| `LLM_PROVIDER`                                                        | Free-text label, recorded in paradata only.                                                                                                                                                                         |
| `LLM_LANGUAGES`                                                       | Comma-separated ISO codes advertised by `supported_languages()` (empty → `[]`).                                                                                                                                     |
| `LLM_MIN_INTERVAL_S` / `LLM_MAX_RETRIES` / `LLM_BACKOFF_BASE_S`       | Throttle/retry knobs (mirror the `LINDAT_*` ones).                                                                                                                                                                  |
| `LLM_GUARD_MIN_CHARS` / `LLM_GUARD_MIN_RATIO` / `LLM_GUARD_MAX_RATIO` | OCR-faithfulness guard thresholds.                                                                                                                                                                                  |

**OCR-faithfulness guardrails** (mirroring `translator.py`'s fail-loud design): temperature
0; a strict system prompt (*translate only; preserve numbers/codes/line structure; do not
correct garbled OCR tokens*); and a post-check that raises `TranslationError` on an empty
response or an implausible output/input length ratio — so a hallucinated or truncated
chunk **skips the file** instead of corrupting the archive.

**Glossary** (`supports_glossary = True`): when a vocabulary CSV is supplied, the term
pairs present in each chunk are injected into the prompt, so terminology is handled
natively and Tag-and-Protect/UDPipe is **not** used for this backend.

Retry/throttle and chunking are the shared `processors/http_retry.py` and
`processors/chunking.py` helpers (same code path as the LINDAT client, so they cannot drift
apart).

### 3.3 `ct2` — `CT2Translator` (Phase 3, design + scaffold)
CTranslate2 int8 self-host (`processors/ct2_translator.py`), targeting **EuroLLM-1.7B/9B**
or **MADLAD-400** (both Apache-2.0). Not registered by default so importing `backend.py`
never pulls in `ctranslate2`/`sentencepiece`; enable via `requirements-ct2.txt` and the two
commented lines in `backend._ensure_registry`.

## 5. Licensing recipe

Output licence is **computed per run** from the components actually exercised
(`para_licenses.resolve_effective_license`, fed by `para_config.txt` and
`ParadataLogger.log_component`). `main.py` records the components a backend reports via
`license_components()`.

Three non-obvious traps make a run non-commercial even when the *translation* model is
permissive:

1. **FastText language-ID is CC BY-NC 4.0** and is **on by default** (`config.txt:
   source_lang = auto`). A permissive output needs an explicit `--source_lang` (or a
   permissive detector) so FastText is never logged.
2. **`amcr_vocab` / `teater_data` are CC BY-NC 4.0** — i.e. the glossary itself is NC. A
   permissive output needs a permissive/empty glossary.
3. The translation model must itself be permissive.

**Permissive recipe:** `--backend ct2` with an Apache-2.0 model (EuroLLM / MADLAD), **plus**
explicit `--source_lang` (no FastText), **plus** a permissive/empty vocabulary → output free
of CC-BY-NC components.

> `llm_api` deliberately carries an unrecognised licence string in `para_config.txt`
> ("LLM provider ToS"): the output terms of an arbitrary free LLM API are provider-specific
> and not a CC/OSS licence, so `para_licenses` treats it as maximally restrictive and warns.
> That is the safe default for a prototype; the licensing owner should pin the real terms
> before any non-experimental use.

## 6. Evaluation plan (`eval/bakeoff.py`, Phase 2)

The model-selection deliverable. Compares CUBBITT vs the LLM adapter (± glossary) on a
sample of AMCR metadata + ALTO OCR.

* **Reference-based** (only when references are supplied via `--refs`): chrF / BLEU via
  `sacrebleu`; COMET via `unbabel-comet` if installed.
* **Reference-less** (the common archival case — OCR text usually has no gold translation):
  COMET-QE / LLM-judge *(optional)* plus heuristic QE — number/date/code preservation rate,
  empty-output rate, and output/input length-ratio distribution.
* **Terminology hit-rate**: fraction of expected glossary targets present in the output, from
  the vocabulary CSV.
* **OCR-robustness spot check**: does the backend hallucinate or "correct" garbled tokens?
* **Pairwise agreement**: character-level similarity between backends as a divergence signal.

Output is a per-segment + summary CSV. Optional deps live in `requirements-eval.txt`; the
harness degrades gracefully when `sacrebleu`/`comet` are absent.

## 7. Adding a new backend

1. Implement the Protocol (`name`, `supports_glossary`, `translate`,
   `supported_languages`) **and** the pipeline-compat surface (`vocabulary`,
   `reset_protected_count`, `protected_count`, ideally `license_components`).
2. Reuse `processors/http_retry.request_with_retry` (network resilience),
   `processors/chunking.chunk_text` (long inputs) and `processors/vocab.load_vocabulary`
   (glossary CSV) rather than re-implementing them.
3. Register it in `backend._ensure_registry` (lazy import to keep optional heavy deps out
   of the base install).
4. Add its component(s) to `para_config.txt` and return them from `license_components`.
5. Add a `tests/test_*_backend.py` mirroring `tests/test_backend.py` (mock the HTTP client;
   no network).
